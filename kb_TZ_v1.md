
# Техническое задание: Микросервис управления базами знаний (Knowledge Base Service)

## 1. Общие сведения

**Наименование системы:** Knowledge Base Service (KBS)
**Основание для разработки:** Необходимость централизации логики работы с векторными хранилищами, повышения безопасности (изоляция данных клиентов), улучшения качества поиска на русском языке и упрощения эксплуатации.

## 2. Обзор предметной области

### 2.1. Ключевые сущности и связи

- **Клиент (Tenant)** — учётная запись на платформе, владелец AI-агентов и баз знаний.
- **AI-агент** — созданный клиентом интеллектуальный агент, взаимодействующий с пользователями.
- **База знаний (БЗ)** — коллекция документов клиента, загруженных на платформу и векторизованных для семантического поиска.

**Связи между сущностями:**

```
Клиент (Tenant)
    ├── 1:N → AI-агенты
    ├── 1:N → Базы знаний
    │
    └── AI-агент ←→ Базы знаний (многие ко многим)
         ├── Пассивный (традиционный) RAG: БЗ подключаются к агенту,
         │    расширяя контекст диалога с LLM. Реализовано в платформе.
         │
         └── Agentic RAG (предлагаемый): LLM через механизм tool calling
              самостоятельно обращается к retrieval tool для поиска в БЗ.
```

### 2.2. Имеющийся функционал (текущее состояние)

**Реализовано на платформе:**
- Создание и управление AI-агентами.
- Создание и наполнение баз знаний.
- Подключение нескольких БЗ к AI-агенту для пассивного (традиционного) RAG — контекст расширяется релевантными чанками из подключённых БЗ при каждом вызове LLM.
- Эндпоинты для работы с БЗ: `/api/v1/embed`, `/api/v1/query`, `/api/v1/filter`, `/api/v1/health`.

**Технические особенности текущей реализации:**
- Векторное хранилище: набор Chroma-баз, одна БД на одну БЗ клиента (отдельная директория файловой системы).
- Эмбеддер: `text-embedding-ada-002` (не оптимален для русского языка).
- Поле «ID векторного хранилища в OpenAI» предусмотрено в интерфейсе платформы, но **не используется**.
- Эндпоинт `/api/v1/filter` (фильтрация чанков по метаданным) предусмотрен, но **не используется**.
- **Проблема безопасности:** при обращении к эндпоинтам доступа к БЗ **не проверяется принадлежность БЗ клиенту**, в рамках сессии которого осуществляется запрос.

### 2.3. Требуемые доработки

1. **Вынос KBS на отдельный сервер:** Выделение всей механики работы с базами знаний в независимый микросервис.
2. **Обеспечение безопасности данных клиентов:** Гарантированная изоляция БЗ разных клиентов. Клиент ни при каких обстоятельствах не должен получить доступ к БЗ, привязанной к чужому аккаунту.
3. **Реализация механизма Agentic RAG:** Поддержка retrieval tool через механизм tool calling, позволяющий LLM самостоятельно осуществлять поиск в БЗ клиента.
4. **Бесшовная миграция:** План перехода с текущей инфраструктуры Chroma на новую без прерывания обслуживания.
5. **Простота администрирования и масштабирования:** Обеспечение простоты стандартных DevOps/DBA операций, мониторинг, контейнеризация.

### 2.4. Предложения по реализации Agentic RAG

Механизм Agentic RAG предполагает, что LLM-агент самостоятельно принимает решение о необходимости поиска в БЗ и вызывает соответствующий инструмент (retrieval tool).

**Предлагаемый подход: описание доступных БЗ в системном промпте агента.**

В системный промпт AI-агента добавляется описание доступных баз знаний и инструкция по использованию retrieval tool:

```
Вы имеете доступ к следующим базам знаний:
- "Документация по продукту" (base_id: abc-123): содержит руководства пользователя и техническую документацию.
- "База знаний поддержки" (base_id: def-456): содержит ответы на частые вопросы и инструкции по решению проблем.

Для поиска информации в базах знаний используйте функцию search_knowledge_base.
Полное описание параметров функции и примеры вызова приведены в json-описании search_knowledge_base.
```

**Примечание:** Параметры вызова `search_knowledge_base` не дублируются в системном промпте. LLM должна руководствоваться формальным json-описанием функции, составленным в соответствии с требованиями MCP (Model Context Protocol).

**Преимущества подхода:**
- LLM получает семантическое описание содержимого БЗ, что позволяет агенту принимать осмысленные решения о том, в какой БЗ искать.
- Клиент может редактировать описание БЗ для более точного управления поведением агента.
- Отсутствие дублирования параметров между системным промптом и json-описанием функции снижает риск рассогласования.

## 3. Выбор инструмента векторного хранилища

### 3.1. Критерии сравнения

Учитывая требования к мультитенантной изоляции, простоте эксплуатации, поддержке кастомных эмбеддеров и минимизации дополнительных компонентов инфраструктуры, проведён анализ трёх кандидатов.

**Ключевая операция, критичная для NoCode-платформы:** регистрация нового тенанта (создание первой БЗ клиента) и создание новых БЗ существующим клиентом. Эта операция должна быть быстрой, атомарной и не требовать сложных административных действий.

| Критерий | Chroma (текущий) | Qdrant | Weaviate |
| :--- | :--- | :--- | :--- |
| **Регистрация нового тенанта / новой БЗ** | Создание отдельной директории, инициализация SQLite БД. Медленно, не атомарно. | **Мгновенно.** Создание коллекции — лёгкая операция, не требующая изменения глобального состояния. | **Сложнее.** Требует создания класса (schema) или тенанта в multi-tenancy конфигурации. |
| **Мультитенантность** | Отсутствует нативно. Реализуется через отдельную физическую БД на диске для каждой БЗ. | **Нативная.** Разделение данных через отдельные коллекции. Гарантированная изоляция. | **Нативная.** Поддержка multi-tenancy через отдельные классы или тенантов. |
| **Гибкость метаданных** | Базовая (плоские структуры, ограниченные операторы фильтрации). | **Schema-flexible.** Payload поля не требуют предварительного объявления. | **Schema-first.** Все поля требуют предварительного объявления в схеме класса. |
| **Поддержка разной размерности векторов** | В текущей реализации отсутствует. | **Полная.** Каждая коллекция имеет свою размерность. | Поддерживается, но требует управления классами. |
| **Простота эксплуатации (DevOps)** | **Низкая.** Встроенная БД (SQLite/DuckDB), не имеет нативного Raft-протокола. Бэкап — копирование файлов. Мониторинг беден. | **Высокая.** Написан на Rust. Поставляется одним бинарником. Встроенная репликация и снапшоты. Отличная Prometheus-метрика. Потребляет мало памяти. | **Средняя.** Требует Java, тяжеловесен. Сложнее в тонкой настройке. |
| **Бэкенды хранения** | SQLite / DuckDB (встроенные). | RocksDB (встроенный). | **На выбор:** RocksDB, Cassandra, S3, GCS, Azure Blob Storage. Наиболее употребительные: **RocksDB** (локально), **S3** (облачно), **Cassandra** (распределённо). |
| **Нагрузка на файловую систему** | Отдельные файлы SQLite. Проблемы с блокировками, фрагментацией. | Коллекции разбиты на иммутабельные сегменты. Общий WAL. mmap-доступ без блокировок. | Зависит от выбранного бэкенда. |
| **Поддержка asyncio** | Условная. LangChain предоставляет async-обёртки (`asearch`, `aadd_documents`), но под капотом — синхронный SQLite, обёрнутый в `run_in_executor()`. При высоких нагрузках упирается в GIL и блокировки SQLite на уровне файла БД. | **Нативная.** Async-клиент (`AsyncQdrantClient`) на gRPC/REST. Сервер на Rust с async I/O. Параллельные запросы к разным коллекциям через `asyncio.gather()` без потоков и блокировок. | **Нативная.** Async Python-клиент. Поддерживает gRPC и GraphQL. |
| **Производительность** | Деградирует при большом количестве файлов БД. | **Лучшая в классе.** Оптимизирован под миллиарды векторов. | Высокая, но выше накладные расходы JVM. |
| **Возврат relevance в результатах** | Да. | **Да.** Поддержка Cosine, Dot Product, Euclidean с настраиваемым порогом. | Да. |

### 3.2. Анализ ключевого различия: регистрация тенанта

Хотя фильтрация по метаданным в платформе пока не используется, критичным остаётся различие в управлении схемой данных, которое напрямую влияет на операцию регистрации нового тенанта и создания новых БЗ.

**Weaviate (Schema-first):**
- Создание нового класса или тенанта — операция, затрагивающая глобальное состояние кластера.
- При добавлении клиента с новым типом метаданных требуется изменение схемы класса через Schema API.
- В multi-tenancy режиме требуется явное создание и активация тенанта перед загрузкой данных.

**Qdrant (Schema-flexible):**
- Создание коллекции — изолированная операция, не затрагивающая другие коллекции.
- Поля payload могут быть произвольными без предварительного объявления.
- Мгновенное создание коллекции без дополнительных административных шагов.

### 3.3. Итоговое предложение: Qdrant

**Обоснование:**
1. **Изоляция и безопасность:** Одна коллекция на базу знаний клиента. Физическое разделение гарантирует, что клиент никогда не получит доступ к чужой БЗ. В отличие от текущей реализации, KBS будет проверять принадлежность БЗ клиенту при каждом запросе.
2. **Простота регистрации новых клиентов и БЗ:** Создание коллекции в Qdrant — мгновенная и изолированная операция, не требующая изменения глобальной схемы. Не нужно управлять классами или тенантами на уровне хранилища.
3. **Schema-flexible для будущих расширений:** Хотя фильтрация по метаданным сейчас не используется платформой, архитектура Qdrant не накладывает ограничений на структуру payload, что позволит внедрить эту функцию без миграции данных.
4. **Кастомные эмбеддеры:** Каждая коллекция может иметь свою размерность векторов, что позволяет клиентам использовать собственные эмбеддеры.
5. **Простота DevOps:** Single binary, иммутабельные сегменты, общий WAL, mmap-доступ — минимальная нагрузка на ФС и ОС. Встроенная репликация и снапшоты. Отличная Prometheus-метрика.
6. **Минимизация инфраструктуры:** Конфигурации БЗ хранятся в мета-коллекции Qdrant — не требуется дополнительная СУБД.

## 4. Архитектура данных

### 4.1. Основные коллекции Qdrant

Для каждой базы знаний клиента создаётся отдельная коллекция:
- **Имя коллекции:** `kb_{tenant_id}_{base_id}`
- **Размерность вектора:** Определяется конфигурацией эмбеддера (стандартный или кастомный).
- **Метрика дистанции:** Cosine Distance (рекомендуемая для текстовых эмбеддингов).
- **Payload (метаданные точки):**
    - `text`: string — исходный текст чанка.
    - `doc_id`: UUID исходного документа.
    - `chunk_index`: integer — номер чанка в документе.
    - `created_at`: datetime — временная метка загрузки чанка.


### 4.2. Мета-коллекция для хранения конфигураций

Для минимизации количества компонентов инфраструктуры конфигурации баз знаний хранятся в самом Qdrant в специальной мета-коллекции.

- **Имя коллекции:** `kb_metadata`
- **Размерность вектора:** 1 (фиктивный вектор `[0.0]`, векторный поиск не используется)
- **Метрика дистанции:** Dot Product (значение не имеет значения, поиск только по payload)
- **Параметры HNSW:** `m=1` (минимальный граф, 16 МБ рёбер на 1М точек)
- **Оптимизация хранения:**
    - `on_disk=True` — HNSW граф на диске.
    - `memmap_threshold=1` — векторы сразу на диск через mmap.
    - `full_scan_threshold=1` — всегда полное сканирование, граф игнорируется.
- **Payload (конфигурация БЗ):**
    - `type`: "kb_config"
    - `tenant_id`: UUID клиента-владельца.
    - `base_id`: UUID базы знаний.
    - `collection_name`: имя основной коллекции в Qdrant.
    - `documents`: словарь `{doc_id: doc_name}`, отражающий все документы, включённые в коллекцию.
    - `embedding_config`: json с параметрами эмбеддера:
        - `type`: "standard" | "custom"
        - `model_name`: string
        - `dimension`: integer
        - `url`: string (только для custom)
    - `disk_usage`: integer объем дискового пространства, занятого коллекцией
    - `created_at`: datetime
    - `updated_at`: datetime
- **Индексы payload:** Создаются на поля `tenant_id` и `base_id` (тип KEYWORD) для быстрой фильтрации.

**Оверхед мета-коллекции на 1 000 000 БЗ:**
- Диск: ~60-100 МБ
- RAM (page cache ОС): ~10-20 МБ


## 5. Функциональные требования (API)

### 5.1. Способы передачи Tenant ID

Рассмотрены пять возможных способов идентификации тенанта в запросах:

| Способ | Пример | Плюсы | Минусы |
|--------|--------|-------|--------|
| **Параметр запроса** | `tenant_id=uuid` | Простота реализации. | Смешивание идентификации с бизнес-параметрами. Не соответствует REST-принципам. |
| **Заголовок запроса** | `X-Tenant-ID: uuid` | Чистое разделение с бизнес-логикой. Удобно для middleware. | Не виден в URL, сложнее отладка. Не кешируется на уровне HTTP. |
| **URL как path** | `/api/v1/kb/{tenant_id}/...` | Явная адресация ресурса. Виден в логах. Соответствует REST. Автоматическое логирование прокси. | Более длинные URL. |
| **URL как поддомен** | `https://{tenant_id}.kbs...` | Максимальная изоляция. Удобно для SaaS. | Сложнее в настройке (DNS, сертификаты). Избыточно для внутреннего микросервиса. |
| **Извлечение из JWT** | `tenant_id` в claims токена | Не требует явной передачи. Исключает возможность подмены. Упрощает API. | Требует валидации JWT на каждом запросе. Для сервисных операций (миграция, администрирование) может потребоваться дополнительный способ указания клиента. |

Рекомендуемый вариант — извлечение `tenant_id` из JWT-токена (пятый способ), так как это исключает возможность подмены идентификатора клиента.

### 5.2. Аутентификация

Тип токена в заголовке `Authorization` — **JWT (Bearer token)**. JWT должен содержать в claims:
- `tenant_id` — идентификатор клиента.
- `kb_quotas` — информация о лимитах на объем хранения.
- `exp` — срок действия токена.

### 5.3. Эндпоинты (первый этап)

На первом этапе поддерживаются только минимально необходимые эндпоинты.

#### Создание/наполнение базы знаний
```
POST /api/v1/kb/embed
```
**Заголовки:**
- `Authorization: Bearer <JWT>`
- `X-Embedding-API-Key: <key>` (API-ключ эмбеддера)

**Тело запроса:**
```json
{
    "openai_api_key": "(опционально) API-ключ эмбеддера - для совместимости с текущей версией",
    "base_id": "uuid (опционально, если не указан — создаётся новая БЗ)",
    "prefix": "kb_",
    "chunk_size": 1000,
    "chunk_overlap": 200,
    "separator": "--- BLOCK START ---",
    "file_types": ["список", "допустимых расширений", "имен файлов", "для загрузки"],
    "files_urls": ["(опционально)", "список", "url", "файлов", "для загрузки"],
    "yandex_disk_folder": "(опционально) url публичной папки для загрузки",
    "text": "(опционально) дополнительный текст для загрузки",
    "url": "(опционально) - .html для рекурсивного извлечения текста, если не задан text",
    "record_id": "temp_record (опционально для совместимости)",
    "email": "unknown (опционально для совместимости)",
    "embedder_type": "platform",
    "embedder_url": "(опционально) для embedder_type: custom",
    "embedder_model": "text-embedding-3-small",
    "embedder_dim": 1536
}
```

**Логика обработки:**
1. Валидация JWT, извлечение tenant_id.
2. Проверка принадлежности БЗ клиенту (если `base_id` указан).
3. Если БЗ новая — создание записи в мета-коллекции `kb_metadata` и основной коллекции в Qdrant.
4. Чанкование документов.
5. Векторизация чанков указанным эмбеддером.
6. Сохранение векторов и payload в основную коллекцию.
7. Обновление словаря `documents` в мета-коллекции.

**Ответ:**
```json
{
    "base_id": "emb_db:uuid",
    "chunks": 42,
    "tokens": 15000,
    "cost_usd": 0.03,
    "status": "success"
}
```

#### Семантический поиск
```
POST /api/v1/kb/query
```
**Заголовки:**
- `Authorization: Bearer <JWT>`
- `X-Embedding-API-Key: <key>` (API-ключ эмбеддера)

**Тело запроса:**
```json
{
    "openai_api_key": "(опционально) API-ключ эмбеддера - для совместимости с текущей версией",
    "base_id": "[emb_db:]uuid",
    "base_ids": ["список", "base_id", "для", "Agentic RAG"],
    "question": "текст запроса",
    "k": 3,
    "relevance_threshold": 0.3
}
```

**Примечания:**
- `base_ids` — LLM может запросить параллельный поиск по нескольким БЗ 
- `relevance_threshold` — порог релевантности в нормализованном диапазоне 0-1.

**Логика:**
1. Валидация JWT, извлечение tenant_id.
2. Поиск по `base_id` в коллекции `kb_metadata` и проверка принадлежности БЗ клиенту.
3. Векторизация запроса соответствующим эмбеддером.
4. Поиск в соответствующей коллекции.

**Ответ:**
```json
{
    "results": [
        {
            "base_id": "emb_db:uuid",
            "id": "chunk_id",
            "page_content": "текст чанка",
            "metadata": {
                "doc_id": "идентификатор исходного документа",
                "source": "имя исходного документа",
                "page": "- для совместимости",
            },
            "relevance": 0.92,
            "raw_distance": 0.16,
        }
    ],
    "tokens": 150,
    "cost_usd": 0.0003,
    "status": "success"
}
```

**Примечания к relevance:**
- `relevance` — нормализованная релевантность 0-1 (1 + cos)/2. 
- `raw_distance` — исходное значение косинусного расстояния из векторной БД (для отладки).

#### Удаление базы знаний

```
**Заголовки:**

**Тело запроса:**

}
```
**Логика:**

2. Проверка принадлежности БЗ клиенту.
3. Удаление коллекции.

## 6. Поддержка кастомных эмбеддеров

### 6.1. Конфигурация
Клиент может указать собственный сервис эмбеддинга для своей БЗ:
- `url` — эндпоинт, совместимый с OpenAI Embeddings API (`POST /v1/embeddings`).
- `api_key` — ключ доступа, передаётся в заголовке `X-Embedding-API-Key` при каждом запросе (не сохраняется на сервере).
- `dimension` — размерность вектора (определяется автоматически тестовым запросом).

### 6.2. Создание коллекции под кастомный эмбеддер
При создании БЗ с кастомным эмбеддером:
1. KBS выполняет тестовый запрос к эмбеддеру клиента для определения размерности.
2. Создаётся коллекция Qdrant с соответствующей размерностью.
3. Размерность сохраняется в мета-коллекции.

### 6.3. Миграция при смене эмбеддера
При изменении конфигурации эмбеддера для существующей БЗ:
1. Создаётся новая коллекция с новой размерностью.
2. Все существующие чанки ревекторизуются новым эмбеддером.
3. Старая коллекция удаляется.
4. Обновляется запись в мета-коллекции.

## 8. План миграции (Стратегия «Ленивая миграция» с проксированием)

### Фаза 1: Деплой KBS с проксированием (2-3 недели)
1. Разворачивается микросервис KBS и инстанс Qdrant.
2. **Все запросы платформы переключаются на эндпоинты KBS**.
3. **Логика KBS:**
   - При получении запроса KBS проверяет наличие коллекции в Qdrant через мета-коллекцию `kb_metadata`.
   - **Если коллекция найдена** → KBS обрабатывает запрос самостоятельно (Qdrant).
   - **Если коллекция не найдена** → KBS **проксирует** запрос на старый эндпоинт платформы (Chroma), выступая в роли обратного прокси.
4. Сервис создания баз знаний модифицируется: все **новые** базы создаются через KBS → Qdrant.
5. Для клиента и платформы переход полностью прозрачен — все эндпоинты сохраняют обратную совместимость.

**Схема проксирования:**
```
Платформа → KBS → [мета-коллекция: коллекция есть?]
                    ├── Да → Qdrant → ответ
                    └── Нет → Старый эндпоинт Chroma → ответ
```

### Фаза 2: Миграция накопленных данных
1. Разрабатывается утилита `migration-worker`.
2. Администратор выбирает старую базу (`base_id`) и запускает задачу миграции.
3. Процесс миграции одной БЗ:
    - Чтение всех чанков из Chroma DB с диска.
    - Отправка в **новый** эмбеддер (альтернвтива - оставлять на `text-embedding-ada-002`).
    - Вставка векторов и payload в новую коллекцию Qdrant.
    - Создание записи в мета-коллекции `kb_metadata`.
    - Валидация целостности (сравнение количества чанков).
    - После успешной миграции KBS автоматически перестаёт проксировать запросы для этой БЗ и начинает обслуживать её из Qdrant.
    - Старая директория Chroma помечается как архивная.
4. Миграция выполняется постепенно, в плановом порядке, без прерывания обслуживания.

### Фаза 3: Отключение Chroma
1. Проверка, что все базы перенесены (все записи есть в мета-коллекции `kb_metadata`).
2. Отключение логики проксирования в KBS.
3. Удаление кода работы с Chroma из монолита.
4. Удаление файлового хранилища Chroma (архивные директории).

## 11. Архитектурные уточнения и принятые решения (v1.2)
*Данный раздел фиксирует технические решения, принятые на этапе проектирования и реализации, которые конкретизируют или оптимизируют исходное ТЗ. Все решения реализованы в финальной версии `main.py` и конфигурационных файлах.*

### 11.1. Пайплайн чанкования (уточнение к разд. 5.3)
| Параметр | Было в ТЗ | Стало | Обоснование |
|----------|-----------|-------|-------------|
| `chunk_size` / `overlap` | `1000` / `200` | `500` / `80` (символы, `len()`) | Повышение семантической плотности чанков, снижение "размытия" контекста. |
| Сплиттер | Не указан | Каскад: `MarkdownHeaderTextSplitter` → `RecursiveCharacterTextSplitter` | Сохранение структурных границ документов. |
| Контекст заголовков | Не указан | Автоматическая инъекция пути `H1 > H2 > H3\n` в начало каждого чанка | Dramatically улучшает Recall@K для технических мануалов и спецификаций. Эмбеддер получает явный иерархический контекст. |
| `strip_headers` | Не указан | `True` в `MarkdownHeaderTextSplitter` | Исключает дублирование заголовков в тексте чанка после инъекции пути. |

### 11.2. Учёт токенов и биллинг (уточнение к разд. 7.6)
- **Было:** Подсчёт через библиотеку `tiktoken`.
- **Стало:** Прямое извлечение `usage.total_tokens` из ответа OpenAI-compatible API (`AsyncOpenAI.embeddings.create`).
- **Кастомные эмбеддеры:** `tokens: 0`, `cost_usd: 0.0`. Платформа не берёт на себя биллинг за сторонние API-ключи клиента.
- **Обоснование:** Исключение расхождений между локальной эвристикой и реальным списанием провайдера. Повышение прозрачности биллинга.

### 11.3. Нормализация релевантности и пороговая фильтрация
- **Базовая формула:** `relevance = (cosine_score + 1.0) / 2.0` (приведение к диапазону `[0.0, 1.0]`).
- **Калибровка:** Параметры `calibration_params` (`min`, `max`) хранятся в `kb_metadata.embedding_config`. При поиске применяется линейный маппинг на единую шкалу, что позволяет использовать **один `relevance_threshold`** для всех моделей.
- **Threshold logic:** Параметр `relevance_threshold` в запросе опционален. Если `null` — фильтрация не применяется (возвращается топ-`k`). Если задан — применяется к *калиброванному* значению.
- **Мониторинг:** Распределение `score` экспортируется в Prometheus. Алерты на дрейф калибровки настраиваются отдельно (см. 11.7).

### 11.4. Обработка запросов и таймауты
- **Batch-эмбеддинг:** Чанки векторизуются пачками по 128 текстов за один API-вызов. Сокращает сетевые round-trips и ускоряет ответ `/embed` в 5–10×.
- **Параллельный поиск:** При указании `base_ids` поиск по каждой БЗ выполняется параллельно (`asyncio.gather`). Эмбеддинг запроса вычисляется один раз на группу `base_ids` с одинаковой конфигурацией.
- **Таймауты:** Вебхуки/callback не используются (сохраняется контракт API). Длительные операции обрабатываются через `--timeout-keep-alive 600` в Uvicorn + обязательную настройку `proxy_read_timeout 600s` на reverse proxy (Nginx/K8s Ingress).

### 11.5. Инфраструктурные и ранжирующие решения
- **RRF (Reciprocal Rank Fusion):** На первом этапе **не применяется**. Ранжирование идёт строго по калиброванной `relevance`. Добавление RRF возможно как опциональный пост-процессор при подтверждённом систематическом смещении рангов между разнородными моделями.
- **Python Runtime:** Образ базируется на `python:3.10-slim`. Фичи 3.11/3.12 не задействованы, что упрощает совместимость с корпоративными CI/CD и снижает размер образа.
- **Миграция (разд. 8 ТЗ):** Проксирование на Chroma реализуется через проверку наличия `collection_name` в `kb_metadata`. При отсутствии записи → fallback на legacy-эндпоинт платформы. Логика прокси выносится в middleware KBS.


### 11.6. Автокалибровка на реальных данных (первый `/embed`)
| Аспект | Реализация |
|--------|-----------|
| **Триггер** | Первый успешный `/embed` для новой `model_name` |
| **Метод** | Выборка 100 векторов → попарные косинусы (~5000 пар) → перцентили 2% и 98% → `[c_min, c_max]` |
| **Защита от вырождения** | Если `c_max - c_min < 0.1` → расширение диапазона на `±0.05` |
| **Хранение** | Запись в коллекцию `kb_models` с флагом `is_calibrated: true` |
| **Fallback** | Если автокалибровка не успела выполниться → базовая нормализация `(cos+1)/2` |

**Преимущества:**
- ✅ Калибровка отражает семантику реальных данных клиента, а не синтетических тестов.
- ✅ Не требует отдельного golden-set или ручного вмешательства.
- ✅ Выполняется асинхронно (`asyncio.to_thread`), не блокируя ответ `/embed`.

### 11.7. Мониторинг дрейфа калибровок
| Метрика | Тип | Как считается | Порог алерта | Назначение |
|---------|-----|---------------|--------------|------------|
| `kbs_drift_psi` | Gauge | PSI между текущим распределением и baseline (10 бакетов) | `> 0.1` warning, `> 0.2` critical | Детекция сдвига распределения после обновления модели провайдером |
| `kbs_score_mean` | Gauge | Скользящее среднее `raw_cosine` (окно 10 000) | Резкое падение `> 15%` за час | Мониторинг общей «уверенности» модели |
| `kbs_score_out_of_range_ratio` | Gauge | Доля скоров вне `[c_min, c_max]` | `> 0.05` (5%) | Сигнал о необходимости перекалибровки |

**Ключевые особенности:**
- ✅ **Не требует фидбека**: работает только на сырых скорах из Qdrant.
- ✅ **Baseline сохраняется в `kb_models`**: переживает рестарты подов.
- ✅ **In-memory rolling window**: `deque(maxlen=10000)` + `numpy.histogram` → нулевой I/O оверхед на запросах.
- ✅ **Асинхронный расчёт**: обновление метрик происходит в фоне, не блокируя event loop.

### 11.8. Коллекция `kb_models`: централизованный реестр эмбеддеров
| Аспект | Реализация |
|--------|-----------|
| **Назначение** | Хранение калибровок, baseline-распределений и метаданных для всех моделей эмбеддинга |
| **Имя точки (`id`)** | `{clean_model_name}` (например, `openai_text_embedding_3_small`) |
| **Индексы** | `model_name` (keyword) — для быстрого поиска и админ-интерфейсов |
| **Вектор** | 1D `[0.0]`, метрика `DOT` (поиск только по payload) |
| **HNSW-конфиг** | `m=1`, `on_disk=True`, `full_scan_threshold=1` — оптимизация под точечные `retrieve` |

### 11.10. Проверка квоты хранения (`tenant_quota_kb`)
| Аспект | Реализация |
|--------|-----------|
